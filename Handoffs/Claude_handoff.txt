RSB is:

* a disciplined sportsbook analytics / +EV simulation project
* a research-backed model, simulation, backtest, and review system
* eventually intended to help build data-backed picks and parlays with reasoning behind every leg

RSB is not:

* a lock generator
* a random pick generator
* a parlay generator yet
* a frontend/API product yet
* a live betting bot
* an auto-betting system

Long-term vision:
The final version of RSB should help build research-backed, data-backed picks and parlays.

The goal is to move away from casual assumptions like:
"Mbappé is a star, so he'll score."

Instead, RSB should eventually justify every candidate leg using structured evidence such as:

* model probability
* sportsbook implied probability
* estimated edge
* historical form
* xG / shot / chance quality
* opponent profile
* lineup / injury / rotation context
* tactical / formation context
* market semantics
* risk flags
* data quality flags
* correlation risk if used in a parlay

Parlays are part of the eventual long-term scope, but not yet.

Future parlay logic must be:

* data-backed
* correlation-aware
* edge-aware
* risk-adjusted
* capable of rejecting bad slips instead of forcing action

RSB should never become reckless or vibe-based. It should prefer pass/no bet when the edge is weak.

Long-term automation goal:
The user should only need to provide high-level run parameters such as sport, date range,
markets, sportsbooks, and risk preferences. RSB should eventually collect fixtures, pull
sportsbook odds, pull stats/injuries/lineups, build features, run models, calculate implied
probabilities and edge, evaluate candidates, and produce ranked outputs automatically.
Current primitive functions take caller-supplied values only because they are being developed
and tested in isolation. Later internal pipeline modules will supply those values automatically.
The user will never manually input edge values, implied probabilities, or player stats
in the final system.

Long-term self-learning goal:
RSB should eventually learn from its own prediction history through controlled feedback loops.
The intended self-learning cycle is:

1. Make prediction / candidate evaluation (pre-game snapshot)
   - record odds, model probability, implied probability, edge, features, context
2. Ingest final result
   - hit / miss, final score / stat outcome, closing line, market movement
3. Run structured post-result review
   - classify why a leg missed: variance, bad calibration, injury surprise,
     line movement, wrong market semantics, weak data, feature gap, correlation issue
4. Store structured review notes and miss reasons in the database
5. Accumulate history and use it to improve:
   - model calibration
   - feature engineering
   - pass/reject rules
   - market selection rules
   - data quality rules
   - parlay correlation rules

RSB should NOT learn recklessly:
* Bad: "One leg missed — change the model immediately."
* Good: "Track this miss, classify why it happened, accumulate evidence across many
         outcomes, then recalibrate only when patterns are statistically meaningful."

Sports are noisy. A good bet can lose. A bad bet can win. RSB must learn from patterns
over many outcomes, not overreact to individual misses.

The primitives already built — review_taxonomy.py, review_notes.py, odds.py, edge.py,
candidate_evaluation.py, backtest_review.py, ev.py, prop_candidate.py, odds_snapshot.py,
and prop_result.py — are early vocabulary for the future self-learning loop.

RSB NORTH STAR / PRODUCT MINDSET
═══════════════════════════════════════════════════════

RSB's long-term goal is to become an autonomous, research-backed sportsbook analytics system that can identify and rank the best +EV prop/pick candidates.

The desired final product should eventually be able to:

* ingest sportsbook odds and player prop markets
* ingest or connect to player/team stats, injuries, lineups, availability, matchup context, and historical results
* run simulations/models/backtests/calibration checks
* compare model probability against sportsbook implied probability
* calculate EV/edge
* rank candidate props/picks by expected value, confidence, data quality, and risk
* construct analytically coherent parlays from compatible candidates
* reject bad parlays even when individual legs look decent
* explain why each candidate or parlay is worth considering or why it should be passed
* track whether the model has genuine edge through CLV, calibration, and settled results
* prefer no-bet/pass when data quality, calibration, or edge is insufficient

RSB is not intended to stay as disconnected primitives forever. The primitive-by-primitive process is intentional because the final predictor must be disciplined and auditable, not a fake lock generator.

Important distinction:
When a version says it did not add live odds APIs, player prop data, sportsbook integrations, simulations, injuries, lineups, historical data, calibration, recommendations, or ranked picks, that means those items were out of scope for that specific version only. It does not mean those capabilities are banned from the project.

Current development philosophy:

* Build stable primitives first.
* Add schemas/contracts before wiring.
* Add provider ingestion only when internal records can validate it.
* Add ranking/recommendation only after odds, EV, data quality, calibration, and backtest review foundations are strong enough.
* Never call anything a lock.
* Do not force picks.
* The end goal is useful ranked +EV prop/pick candidates, but every step must preserve correctness, testability, and auditability.

EXTENDED PROJECT CONTEXT AND PRINCIPLES
═══════════════════════════════════════════════════════

RSB FINAL GOAL — SINGLES AND PARLAYS
─────────────────────────────────────────────────────

RSB's final product is a disciplined system for:

1. Finding ranked +EV single prop/pick candidates.
2. Rejecting weak, noisy, or no-edge singles.
3. Constructing analytically coherent parlays from compatible candidates.
4. Rejecting bad parlays even when individual legs look decent.
5. Explaining every candidate and parlay with reasoning, data quality, uncertainty,
   and pass conditions.
6. Tracking whether the model has genuine edge through CLV, calibration, and settled results.

PARLAY CONSTRUCTION PHILOSOPHY
─────────────────────────────────────────────────────

RSB must not bundle random +EV singles together and call it a parlay.

Parlay narrative trap to avoid:
  "Team A is better → blowout expected → Team A stars go over → Team B players go under."
  This dies if the game is close, pace is low, starters rest in Q4, foul trouble disrupts
  rotations, or the underdog keeps it competitive.

A game can go: close, one-sided, high pace, low pace, foul-heavy, rotation-heavy,
injury-affected, bench-heavy, or overtime.

A good parlay should not be dead solely because one simple game script fails.
The goal is not parlays immune to all outcomes — that is impossible.
The goal is to avoid cards where every leg depends on the same brittle assumption.

Parlay construction must eventually evaluate:
  individual leg EV and edge
  joint probability
  scenario-adjusted joint EV
  sportsbook parlay price
  same-game / same-player / same-team exposure
  market overlap, contradiction, cannibalization
  duplicate exposure to one player/team/game environment
  fragile shared assumptions
  data quality, calibration strength, line movement sensitivity
  CLV history where available

Correct correlation principle:
  Positive correlation usually increases joint hit probability vs. independent multiplication.
  Negative correlation usually decreases joint hit probability vs. independent multiplication.
  Neither direction is automatically good or bad.
  The target is scenario-adjusted joint EV vs. sportsbook parlay price.
  Same-game parlay books may already include correlation adjustments — RSB must not
  blindly chase correlation.

Structural compatibility vs. empirical correlation:
  Structural detection (same player, same team, same game, overlap, contradiction,
  cannibalization) can be done with rules/schema and belongs in v0.7.x.
  Empirical correlation estimation requires historical outcome data and belongs in
  the v0.4.x-v0.5.x historical/modeling layers. Do not conflate these two concepts.

CLV — CLOSING LINE VALUE
─────────────────────────────────────────────────────

CLV is a first-class long-term proof-of-edge metric.

If RSB consistently identifies candidates whose prices are better than the closing line
over a meaningful sample, the model has genuine edge. If not, wins are likely variance.

RSB should eventually track:
  found price / odds_found_at (on the odds snapshot record, not the candidate record)
  best available price
  closing price and closing line
  whether RSB beat the close
  CLV by market, sportsbook, model version, confidence bucket, data-quality bucket

TIMESTAMP SPLIT
─────────────────────────────────────────────────────

These are different timestamps — do not conflate them:

  created_at:         when RSB created the candidate record
                      Lives on the candidate schema (v0.2.5)

  odds_found_at /     when the specific price was observed from a provider/source
  odds_timestamp:     Lives on the odds snapshot record (v0.2.6)

CLV comparison requires odds_found_at, not created_at. If odds_found_at is not
captured from v0.2.6 onward, CLV calculations will be impossible to retrofit.

DATASET WINDOWS — SPORT-SPECIFIC
─────────────────────────────────────────────────────

Do not hard-code "last two seasons" as a universal rule.

NBA: 2 seasons is a reasonable starting window. Weight current season heavily.
     Role/team/coach changes override raw historical averages.

MLB: 2 seasons is reasonable given large sample sizes. Key context: park factors,
     handedness, batting order, pitcher/batter matchups, bullpen usage, weather.

NFL: 2 seasons can be thin (~34-38 games per team). Weight current season heavily.
     Coaching, QB, scheme, snap share, and target share changes often matter more
     than historical averages.

World Cup / Soccer: "Last two seasons" does not translate cleanly to international
     football. World Cup is every 4 years. Supplement with: qualifiers, UEFA Euros,
     Copa America, AFCON, Asian Cup, Nations League. Apply uncertainty bounds for
     small samples. Club form is relevant context.

Recency weighting should be learned and validated from calibration data, not
permanently hardcoded as fixed constants.

FEATURE HIERARCHY — GENERAL PRINCIPLE
─────────────────────────────────────────────────────

Tier 1 — Core projection drivers (most important):
  minutes / availability / usage / role / starting status / lineup context

Tier 2 — Game environment:
  spread, total, pace, team implied total, blowout risk, rest/travel, weather

Tier 3 — Matchup / tactical / play-type features (model refinement layer):
  pick-and-roll role/coverage, switching, set pieces, defensive scheme,
  play-type distribution, transition frequency, foul tendencies, rim protection

Tier 4 — Research / qualitative context (must be converted to structured inputs):
  injury reports, lineup news, coach quotes, beat reporter notes, rotation changes,
  load management, trade/roster moves, market movement

Tactical/play-type data matters and should be encoded as structured features, not
freeform narrative. It belongs in v0.5.x model refinement, not v0.2.5 schema.

Feature ablation principle:
  Features must earn their place. RSB should eventually test whether each feature
  improves Brier score, log loss, or CLV over a meaningful sample. Features that
  do not improve predictive quality should be demoted or removed.

INJURY REDISTRIBUTION MODELING
─────────────────────────────────────────────────────

When a key player is out, RSB should eventually model usage/minutes redistribution
to teammates using historical lineup-adjusted data.

Example: Brunson out → KAT usage increases, secondary guards take more assist chances,
Knicks offensive efficiency may drop, opponent defensive focus shifts.

Markets lag on secondary player adjustments — this is a first-class edge opportunity.
Must be data-validated, not treated as automatic narrative logic.

RESEARCH LAYER — REAL-TIME REQUIREMENT
─────────────────────────────────────────────────────

Injury news and lineup confirmation often arrive 60-90 minutes before game time.
A single daily batch is not the final automation architecture.

Long-term, RSB should support multiple scan windows:
  Morning: general candidate identification, baseline odds snapshot
  Afternoon: injury/lineup update check, candidate flag refresh
  Pre-game: final lineup confirmation, late line movement, final go/no-go

The pre-game window is the highest-value window for research-based adjustments.

SETTLEMENT BEHAVIOR — SPORT AND BOOK SPECIFIC
─────────────────────────────────────────────────────

Settlement behavior is sportsbook-specific, sport-specific, market-specific, and
sometimes parlay-format-specific. RSB must not assume universal settlement rules.

For player props, settlement varies significantly by book and market:
  Some books/markets require the player to start.
  Some require only participation (any appearance).
  Some void props if the player is inactive or does not start.
  Some settle once the player participates regardless of role.
  Soccer has especially important start/appearance/void-rule differences, but
  NBA/NFL/MLB/NHL also have book-specific participation, inactive, and early-exit rules.

The schema and settlement roadmap must preserve support for fields such as:
  start_required
  participation_required
  void_condition
  settlement_rule
  book-specific settlement metadata

v0.2.5 preserved metadata support where scoped. Full settlement primitives were implemented in v0.2.7.
RSB must not assume any universal settlement behavior across sports, books, or markets.

ACCOUNT HEALTH / EXECUTION CONSTRAINTS
─────────────────────────────────────────────────────

RSB can identify theoretical +EV opportunities, but execution depends on:
  which books the user can access
  whether limits or restrictions exist
  whether the best available line is actually placeable
  odds movement before placement
  book-specific max bet constraints

RSB should not assume every theoretical best price is executable.
This is a product constraint, not a technical one.

DATA BOTTLENECKS — WHAT IS ACTUALLY HARD
─────────────────────────────────────────────────────

The hardest parts of RSB are not EV math. EV math is already built. The hard parts are:
  clean historical player prop data with settlement
  settled prop outcomes over enough samples per market
  closing line history for CLV
  reliable real-time injury/lineup data
  sportsbook/player/team ID normalization across providers
  odds provider coverage and rate limits
  market availability differences by sport/book
  calibration over enough samples
  play-type/tactical datasets, which may be expensive or unavailable

RSB should not pretend it is calibrated or autonomous until the data layer supports
that claim with verified settled results.

PROJECT IDENTITY
═══════════════════════════════════════════════════════

RSB is my disciplined sportsbook analytics / +EV simulation project.

RSB is:

* a research-backed sportsbook analytics project
* a +EV simulation and review system
* a disciplined model/backtest/evaluation framework
* eventually intended to help build data-backed picks and parlays with reasoning behind every leg
* automation-first in its long-term design
* intended to become self-learning through controlled feedback loops

RSB is not:

* a lock generator
* a random pick generator
* a parlay generator yet
* a frontend/API product yet
* a live betting bot
* an auto-betting system
* a prompt-to-pick system
* an LLM-vibe betting assistant
* an uncontrolled self-modifying model

Long-term vision:
The final version of RSB should help me build research-backed, data-backed picks and parlays.

The goal is to move away from casual assumptions like:

"Mbappé is a star, so he'll score."

Instead, RSB should eventually justify every candidate leg using structured evidence such as:

* model probability
* sportsbook implied probability
* estimated edge
* expected value
* historical form
* xG / shot / chance quality
* opponent profile
* lineup / injury / rotation context
* tactical / formation context
* market semantics
* risk flags
* data quality flags
* review taxonomy labels
* structured review notes
* correlation risk if used in a parlay

Parlays are part of the eventual long-term scope, but not yet.

Future parlay logic must be:

* data-backed
* correlation-aware
* edge-aware
* risk-adjusted
* capable of rejecting bad slips instead of forcing action

RSB should never become reckless or vibe-based. It should prefer pass/no bet when the edge is weak, data quality is poor, or the model is not calibrated enough.

═══════════════════════════════════════════════════════
LONG-TERM AUTOMATION GOAL
═══════════════════════════════════════════════════════

The final workflow should not require manual match/team/player/odds/props/stat input.

The user should only provide high-level run parameters, such as:

* sport
* league
* date range / week
* markets
* sportsbooks
* singles vs. number of parlay legs
* risk preference / strictness level

Example future user command:

"Run RSB for NBA player props this week across DraftKings and FanDuel."

or:

"Run RSB for World Cup games this weekend and show the best +EV markets."

RSB should eventually automate:

1. Collecting fixtures / games / slates
2. Collecting sportsbook odds, props, and lines
3. Collecting team/player stats
4. Collecting injuries, lineups, rest, rotation, travel, weather, and contextual signals where available
5. Building features
6. Running simulations/models
7. Producing model probabilities
8. Converting sportsbook odds to implied probabilities
9. Calculating edge
10. Calculating EV when that layer exists
11. Evaluating candidate eligibility/pass reasons
12. Applying risk/data-quality/calibration checks
13. Ranking opportunities
14. Producing a report/dashboard/output
15. Saying "pass/no bet" when the candidate is weak or insufficiently supported

Important clarification:
When current primitive specs say "the caller supplies edge/status/pass_reasons," "caller" means internal RSB code, not the user manually typing those values.

Current primitive functions take caller-supplied values only because they are being developed and tested in isolation. Later internal pipeline/orchestrator modules will supply those values automatically.

Example future automated chain:

odds provider / sportsbook ingestion
→ implied probability

model / simulation
→ model probability

calculate_edge(model_probability, implied_probability)
→ edge

candidate screening / evaluation code
→ status and pass_reasons

build_candidate_evaluation(status, edge, pass_reasons)
→ candidate record

ranking/output layer
→ ranked opportunities or pass/no-bet report

The user should not manually calculate or input edge in the final product.

═══════════════════════════════════════════════════════
LONG-TERM SELF-LEARNING / FEEDBACK LOOP GOAL
═══════════════════════════════════════════════════════

RSB should eventually become self-learning through controlled feedback loops.

After a prediction, candidate leg, or parlay settles, RSB should automatically:

1. Record the original pre-event decision snapshot:

   * fixture/game
   * sport/league
   * market/prop
   * sportsbook
   * line and odds
   * implied probability
   * model probability
   * edge
   * candidate evaluation status
   * pass/reject reasons if applicable
   * available features/context at decision time
   * injury/lineup/rest/rotation/travel/weather context where available
   * model version / simulation version
   * timestamp of the decision snapshot

2. Ingest the final result:

   * hit / miss / push / void
   * final score or player stat result
   * final box score / event result
   * closing line where available
   * market movement where available
   * whether the market moved for or against the model

3. Run structured post-event review:

   * Was the miss likely normal variance?
   * Was the model miscalibrated?
   * Was data quality weak?
   * Was there a missing injury/lineup/rotation signal?
   * Was the market semantics wrong?
   * Was the matchup/context feature incomplete?
   * Did the line move against the model?
   * Was the number stale?
   * Was there unexpected usage/role change?
   * Was there parlay correlation risk?
   * Did the model overvalue a feature?
   * Did the model underweight a known risk?

4. Store structured review notes / miss reasons in the database.

5. Use accumulated history to improve future:

   * model calibration
   * feature engineering
   * data-quality checks
   * candidate evaluation rules
   * pass/no-bet logic
   * market selection
   * parlay correlation rules
   * risk scoring
   * retraining datasets
   * simulation assumptions

Important guardrail:
RSB should not blindly change its model after a single miss. A good bet can lose and a bad bet can win. Self-learning must be controlled, audited, data-backed, and based on patterns across many settled outcomes.

Bad self-learning:

"One leg missed, so immediately change the model."

Good self-learning:

"Record the miss, classify possible causes, add structured review data, and use accumulated evidence for future retraining/calibration."

The long-term architecture should move toward:

prediction database
→ result ingestion
→ post-event review engine
→ miss reason taxonomy
→ calibration reports
→ feature improvement queue
→ controlled retraining/recalibration
→ better future candidate evaluation

Important:
Self-learning should be audited and controlled. RSB should not allow unverified narratives, LLM guesses, or one-off misses to directly mutate model logic. Post-event review can propose hypotheses, but model updates should be based on verified data, sufficient sample size, calibration checks, and explicit versioned retraining.

Structural invariants enforced by build_candidate_evaluation():

* status="candidate"     → pass_reasons must be []
* status="rejected"      → pass_reasons must have at least one validated reason
* status="not_evaluable" → pass_reasons must have at least one validated reason
* Violating these raises CandidateEvaluationValidationError

Operation order inside build_candidate_evaluation():

1. Normalize and validate status
2. Validate edge via _validate_edge()
3. Normalize and validate pass_reasons via validate_pass_reasons()
4. Enforce structural invariant
5. Return dict

Edge validation rules (_validate_edge):

* None → None (allowed on any status)
* bool → reject
* non-numeric → reject
* NaN → reject
* infinity → reject
* edge < -1.0 → reject
* edge > 1.0 → reject
* valid → return float(edge)

Only import: stdlib math (for _validate_edge NaN/inf checks).
No imports from odds.py, edge.py, review_taxonomy.py, review_notes.py, database, or runtime modules.

Core behavior:

calculate_edge(model_probability, implied_probability):

* Validates model_probability using validate_probability() from odds.py
* Validates implied_probability using validate_probability() from odds.py
* Invalid model_probability raises OddsValidationError from validate_probability()
* Invalid implied_probability raises OddsValidationError from validate_probability()
* Does not catch or wrap OddsValidationError
* Returns model_probability - implied_probability as a plain Python float
* Positive edge means model probability is higher than sportsbook implied probability
* Negative edge means model probability is lower than sportsbook implied probability
* Zero edge means they are equal
* Does not round
* Preserves native Python float behavior
* Valid edge range is naturally -1.0 to 1.0 because both inputs must be valid probabilities
